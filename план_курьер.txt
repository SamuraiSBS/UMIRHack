Требования к приложению курьера (ТЗ «Флагман»)

КОНТЕКСТ
========
Из PDF-документа ТЗ необходимо выписать и детально расписать требования к части приложения для курьеров.
Приложение уже частично реализовано — бэкенд готов полностью, три фронтенд-страницы созданы, но не подключены к роутингу.


ТЕКУЩЕЕ СОСТОЯНИЕ КОДА
=======================

Что уже реализовано:

Backend API (backend/src/routes/courier.js, backend/src/routes/orders.js):
  - GET /api/courier/shift         — статус смены
  - POST /api/courier/shift/start  — начало смены
  - POST /api/courier/shift/stop   — завершение смены
  - GET /api/courier/orders        — список заказов курьера
  - GET /api/orders/available      — доступные заказы (без адреса)
  - POST /api/orders/:id/accept    — принятие заказа
  - PATCH /api/orders/:id/status   — обновление статуса (DELIVERING -> DONE)

Три готовые страницы в frontend/src/pages/courier/:
  - ShiftControl.jsx    — управление сменой
  - AvailableOrders.jsx — список доступных заказов
  - ActiveOrder.jsx     — активный заказ

Критическая проблема:
В frontend/src/App.jsx все маршруты курьера (/courier, /courier/orders, /courier/active)
вели на CourierAppDownload (страница скачивания APK), а не на реальные страницы.
Страницы существовали, но были недоступны. ИСПРАВЛЕНО.


ТРЕБОВАНИЯ ИЗ ТЗ (ПО ПУНКТАМ)
================================

1. РЕГИСТРАЦИЯ КУРЬЕРА
----------------------
Что требуется: Курьер должен иметь возможность зарегистрироваться в системе
с указанием зоны доставки.

Детали:
- Стандартная форма регистрации (email, пароль, имя)
- Роль при регистрации: COURIER
- Указание зоны доставки (географическая зона, район, адрес) — опционально,
  т.к. в текущей схеме БД этого поля нет
- Backend: POST /api/auth/register с role: 'COURIER' — уже работает
- Frontend: страница регистрации существует в /login / /register,
  роль COURIER доступна при регистрации

Статус: Частично реализовано


2. НАЧАЛО И ЗАВЕРШЕНИЕ СМЕНЫ
-----------------------------
Что требуется: Курьер должен иметь кнопку «Выйти на смену» и «Завершить смену».
Без активной смены нельзя принимать заказы.

Детали:
- Отображение текущего статуса смены (активна / неактивна)
- Кнопка переключения статуса смены
- При активной смене — показывать, что можно брать заказы
- При неактивной — все кнопки принятия заказов должны быть недоступны
- Backend: POST /api/courier/shift/start и POST /api/courier/shift/stop — реализованы
- Frontend: ShiftControl.jsx — страница создана и подключена к маршруту /courier

Статус: ПОЛНОСТЬЮ РЕАЛИЗОВАНО


3. ПРОСМОТР ДОСТУПНЫХ ЗАКАЗОВ
------------------------------
Что требуется: Курьер видит список заказов, которые можно взять.
До принятия заказа показывается только ограниченная информация.

До принятия заказа показывается:
  - Стоимость заказа (totalPrice)
  - Точка отправки (название бизнеса/ресторана)
  - Список товаров (что доставляется)
  - Расстояние (если реализовано) — опционально

НЕ показывается до принятия:
  - Полный адрес доставки
  - Контактные данные покупателя

Детали:
- Список обновляется автоматически каждые 8 секунд
- Доступны только курьерам с активной сменой
- Backend: GET /api/orders/available — реализован, возвращает данные без адреса
- Frontend: AvailableOrders.jsx — создана и подключена к маршруту /courier/orders

Статус: ПОЛНОСТЬЮ РЕАЛИЗОВАНО


4. ПРИНЯТИЕ ЗАКАЗА И ПРОСМОТР ПОДРОБНОСТЕЙ
-------------------------------------------
Что требуется: Курьер может принять заказ. После принятия открывается полная информация.

После принятия показывается:
  - Полный адрес доставки
  - Контакты покупателя
  - Детальный состав заказа
  - Кнопки смены статуса: «В пути» (DELIVERING) -> «Доставлен» (DONE)

Ограничения:
  - Одновременно можно принять только 1 заказ
    (если уже есть активный — нельзя взять другой)
  - Backend атомарно проверяет гонку при принятии
    (используется updateMany с условием status: CREATED)

Детали:
- Backend: POST /api/orders/:id/accept — реализован
- Backend: PATCH /api/orders/:id/status — реализован (DELIVERING -> DONE)
- Frontend: ActiveOrder.jsx — создана и подключена к маршруту /courier/active

Статус: ПОЛНОСТЬЮ РЕАЛИЗОВАНО


5. ПРОСМОТР ВЫПОЛНЕННЫХ ДОСТАВОК
----------------------------------
Что требуется: Курьер должен видеть историю выполненных им доставок.

Детали:
- Список заказов со статусом DONE, где courierId = текущий курьер
- Показывается: бизнес, сумма, адрес
- Backend: GET /api/courier/orders — реализован, возвращает все заказы курьера включая DONE
- Frontend: в ActiveOrder.jsx есть секция с историей выполненных заказов

Статус: ПОЛНОСТЬЮ РЕАЛИЗОВАНО


ИСПРАВЛЕНИЕ РОУТИНГА (ВЫПОЛНЕНО)
==================================

Файл: frontend/src/App.jsx

Было (все маршруты вели на страницу скачивания APK):
  /courier        -> CourierAppDownload
  /courier/orders -> CourierAppDownload
  /courier/active -> CourierAppDownload

Стало (реальные страницы):
  /courier        -> ShiftControl
  /courier/orders -> AvailableOrders
  /courier/active -> ActiveOrder


ПРОВЕРКА
=========

1. Зайти под аккаунтом courier@demo.com / demo123
2. Перейти на /courier       — открывается панель управления сменой
3. Нажать "Начать смену"     — статус меняется на активный
4. Перейти на /courier/orders — список доступных заказов, обновляется каждые 8 сек
5. Принять заказ             — автоматический переход к активному заказу
6. Обновить статус: ACCEPTED -> DELIVERING -> DONE
7. Завершить смену           — кнопка "Завершить смену"
8. Выполненные заказы        — отображаются в секции истории на странице /courier/active
